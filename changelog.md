## [2026-04-18]

### Гибридная система экстракции цветов и архитектура кэширования

- **Super-Hybrid Color Engine (Node-Vibrant + FastAverageColor):**
  - **11-точечный анализ**: Реализована экстракция 11 цветовых параметров (Dominant, Sqrt, Simple от FAC; полный спектр Vibrant swatches).
  - **Перцептивная точность**: Доминирующим цветом приложения назначен `FAC Dominant`, обеспечивающий наиболее точное соответствие фоновому тону обложки.
  - **Legacy Mode**: Сохранены и выведены в UI вспомогательные поля `vibrantDominant` и `vibrantAccent` для сохранения преемственности дизайна.

- **Исправление цветопередачи (B&W Fix):**
  - **Sharp Normalization**: Внедрена предварительная обработка изображений через `sharp`. Это устранило проблему "розового оттенка" в черно-белых обложках за счет нормализации цветовых каналов перед анализом.

- **Производительность и Кэширование:**
  - **Absolute Cache Strategy**: Реализована логика "кэш прежде всего". Сервер моментально отдает данные из БД без повторной загрузки и анализа изображений.
  - **Admin Force Mode**: Добавлен защищенный флаг `force: true` (доступен только роли `ADMIN`), позволяющий принудительно пересчитать цвета для исправления старых записей.
  - **Session Stability**: Извлеченные цвета теперь синхронизируются с глобальным стором (`updateTrackColors`). Это убрало мерцание и состояние "извлечения" при навигации между треками внутри одной сессии.
  - **Safe DB Recovery**: Обработка ошибок Prisma (P2025) для виртуальных и системных плейлистов, предотвращающая падение сервера при отсутствии записей в БД.

- **UI Компонент ColorPaletteDisplay:**
  - Создана визуальная панель для отладки палитры (доступна на главной).
  - Группировка цветов по алгоритмам (FAC vs Vibrant) с описанием механизмов извлечения.

### Глобальная переработка аудио-движка (Отказ от react-howler)
- **Полный переход на управление через Zustand Store:**
  - *Решение:* Библиотека `react-howler` была признана нестабильной для React 19 / Strict Mode. Вся логика создания и управления инстансом `Howl` перенесена напрямую в `playerStore.ts`.
  - *Принцип работы:* Теперь при клике пользователя на "Play" или "Трек", инстанс `new Howl()` создается **синхронно** внутри обработчика события. Это гарантирует 100% обход ограничений Autoplay в мобильных браузерах и Chrome, так как вызов происходит в том же стеке вызовов, что и жест пользователя.
  - *Предотвращение утечек:* Перед созданием нового трека, старый инстанс принудительно вызывает `.stop()` и `.unload()`. Это исключает появление "зомби-потоков" и ошибку `pool exhausted`.
- **Синхронизация Media Session API:**
  - *Изменение:* Обработчики системных кнопок ОС теперь напрямую вызывают экшены стора (`togglePlay`, `handleNextTrack`). 
  - *Исправление бага перемотки:* Переписана логика `seekto` в ОС. Теперь она использует дебаунс и напрямую обращается к нативному методу `.seek()` инстанса Howl, минуя лишние циклы рендеринга React.
- **Оптимизация Visualizer (AudioMotion):**
  - *Решение:* Поскольку аудио-нода HTML5 теперь управляется стором, визуализатор подключается к ней через мягкий интервал проверки. Это предотвращает краши при быстрой смене треков, так как визуализатор просто "подхватывает" актуальный `HTMLMediaElement` из текущего инстанса Howl.
- **Исправление UI-консистентности:**
  - Унифицированы пропсы компонентов `Button`. Все некорректные значения `view="outline"` заменены на `view="outline-solid"` в `page.tsx`, `AnimatedHeader.tsx` и `PlaylistPageClient.tsx`.

## [2026-03-29]

### Критические патчи движка воспроизведения (Ошибки циклов и утечек памяти)
- **Устранение бага "Maximum update depth exceeded":** 
  - *Причина:* Возникал сложнейший баг синхронного каскада рендеров в `playerStore.ts`. Логика "жесткого самовосстановления" (через инкремент `playerKey`) при смене трека применяла запомненное значение `seek` от старого трека к новому. Если старая позиция превышала длительность нового трека, движок Howler.js моментально (синхронно) бросал событие `onEnd`. Оно сразу же переключало на следующий трек, вызывая новый маунт и новый `onLoad`, зацикливая приложение намертво.
  - *Решение:* Полностью удалена логика форсированного восстановления через `playerKey` и перемотка при инициализации `onLoad`.
- **Устранение "HTML5 Audio pool exhausted":** 
  - *Причина:* Компонент `<ReactHowler>` имел атрибут `key={currentTrack.id + playerKey}`. Из-за вышеописанного бесконечного цикла, React пересоздавал дерево с такой скоростью, что сборщик мусора Howler не успевал уничтожать внутренние инстансы HTMLMediaElement (оставляя "зомби-ноды"). Это приводило к исчерпанию лимита браузера за сотые доли секунды.
  - *Решение:* Удален свойский `key` у компонента `ReactHowler`. Теперь при смене трека компонент не пересоздается "с нуля", а плавно обновляет пропс `src`, что заставляет библиотеку `react-howler` безопасно утилизировать прошлую ноду.
- **Исправление багов Media Session (Двойные потоки):** 
  - *Причина:* При быстрых кликах перемотки в ОС (например, +10 сек) возникала гонка состояний между UI-подписками на прогресс (`useSeekInterval`) и системным API. Некорректно срабатывающий `triggerRecovery` убивал один плеер, но из-за асинхронности движка старая нода не отключалась, и спавнился дубликат аудио.
  - *Решение:* Удален весь механизм Recovery, добавлен надежный механизм Debounce для любых действий перемотки из ОС (с ожиданием 300мс), предотвращающий спам команд `howler.seek()`.
- **Стабилизация `AudioMotionVisualizer`:** 
  - *Решение:* Из визуализатора удалены агрессивные сбросы (вызов `destroy()` и убийство синглтонов). Настроен умный перехват `InvalidStateError`, возникающего в React 18 Strict Mode, с использованием мягкого переподключения логикой `connectInput` вместо жестких перезагрузок всего окружения.

### Исправления и улучшения Media Session API
- **Стабилизация сессии в фоне:** Исправлена критическая проблема исчезновения уведомления плеера на заблокированном экране при нажатии на паузу. Выявлено, что мобильные браузеры (особенно Chrome на Android) сбрасывают сессию Web Audio, если происходит слишком частое обновление состояния через `navigator.mediaSession.setPositionState`. 
- **Отказ от костылей:** Полностью удален `setPositionState` и вспомогательные "тихие" аудио-якоря. Теперь работа сессии синхронизируется только по событиям смены трека и переключения состояния `playing/paused`, что соответствует поведению нативных плееров и обеспечивает стабильность без перехода на HTML5 audio (сохраняя работоспособность визуализатора).

### Архитектура и производительность
- **Zustand рефакторинг:** Полный переход с `PlayerContext` на `zustand`. Реализовано централизованное хранилище `usePlayerStore` для управления состоянием воспроизведения, очередью и метаданными.
- **Оптимизация ререндеров:** Внедрены атомарные селекторы для доступа к часто меняющимся данным (`seek`, `duration`). Это радикально снизило нагрузку на CPU, так как теперь изменения прогресса трека не вызывают перерисовку всего дерева компонентов плеера.
- **Модульная структура:** Исходный код перенесен в папку `src/`. Компоненты плеера декомпозированы на изолированные модули (`PlayerControls`, `MobilePlayer` и др.) с использованием CSS-модулей для предотвращения конфликтов стилей.

### Функциональные изменения
- **Управление звуком:** Реализован компонент `VolumeSlider`. Добавлена поддержка изменения уровня громкости и режима Mute с сохранением состояния в глобальном сторе.
- **Цвета интерфейса:** Логика извлечения доминантных цветов обложек вынесена в хук `useTrackColor` с механизмом кеширования, что устранило мерцание UI при переключении треков.
- **Тексты песен (Lyrics):** Загрузка текстов теперь независима от готовности аудио-ноды (`duration`). Исправлен баг со сбросом позиции скролла при смене композиции.
- **Исправление багов:** Устранено дублирование ключей в списках треков (`TrackList`), вызывавшее ошибки в React-дереве.
